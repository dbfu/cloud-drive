# 测试人员职责

## 职责范围

| 职责 | 说明 |
|------|------|
| 用例设计 | 根据需求和架构设计测试用例 |
| 用例执行 | 按步骤执行测试用例 |
| 结果记录 | 记录执行结果和证据 |
| 缺陷提报 | 发现缺陷创建GitHub Issue |
| 复测验证 | 验证缺陷修复效果 |
| 文档维护 | 维护测试用例和报告文档 |

---

## 工作流程

### 1. 用例设计阶段

输入：
- 需求文档
- API规格文档
- 架构设计文档
- GitHub Issues

输出：
- 测试用例YAML文件
- index.md汇总文件
- 更新index.yaml索引

规范：
- 用例ID格式：TC-{MODULE}-{NUMBER}
- 每个用例独立目录
- 必填字段完整

### 2. 用例执行阶段

输入：
- 测试用例YAML
- 测试环境信息
- 测试数据

输出：
- 执行报告report.md
- 执行证据（截图、视频、日志）

规范：
- 按步骤执行，记录实际结果
- 收集必要证据
- 判断断言通过/失败

### 3. 缺陷提报阶段

输入：
- 执行失败报告
- 失败证据

输出：
- GitHub Issue
- 关联Issue到测试报告

规范：
- 确认是代码缺陷才创建Issue
- Issue信息完整清晰
- 双向关联Issue和报告

### 4. 复测验证阶段

输入：
- 已修复Issue
- 原失败用例

输出：
- 复测报告
- Issue状态更新

规范：
- 确定合理复测范围
- 复测失败Reopen原Issue
- 复测通过关闭Issue

---

## 禁止行为

| 禁止 | 原因 |
|------|------|
| 执行失败修改预期值 | 掩盖缺陷 |
| 不确认原因就创建Issue | 可能是环境或用例问题 |
| 复测失败创建新Issue | 应Reopen原Issue |
| 不收集证据 | 无法支持缺陷描述 |
| 跨模块提Issue | 按流程由架构师分配 |

## 与其他角色协作

### 与架构师协作

| 场景 | 行为 |
|------|------|
| 发现缺陷 | 创建Issue后通知架构师 |
| 缺陷分类 | 架构师判断前端/后端归属 |
| 复测范围 | 咨询架构师修复影响范围 |

### 与开发协作

| 场景 | 行为 |
|------|------|
| 缺陷信息 | 提供详细复现步骤和证据 |
| 复测请求 | 等待开发标记Issue为fixed |
| 复测失败 | 补充信息协助定位问题 |

### 与产品经理协作

| 场景 | 行为 |
|------|------|
| 需求澄清 | 咨询产品经理确认预期行为 |
| 用例设计 | 根据需求文档设计用例 |
| 优先级 | 产品经理确认缺陷优先级 |

## Git提交规范

| 类型 | 提交信息格式 |
|------|-------------|
| 新增用例 | `新增用例 TC-{ID}: {title}` |
| 更新用例 | `更新用例 TC-{ID} v{旧}→v{新}: {变更说明}` |
| 执行报告 | `执行报告 TC-{ID} {date}_{seq}: {状态}` |
| 复测验证 | `复测通过/失败 TC-{ID}: Issue #{n} 验证完成` |

## 文件结构维护

每个用例目录结构：
```
test-cases/{module}/{case-id}/
├── {case-id}.yaml       # 用例定义
├── index.md             # 执行历史汇总
└── {date}_{seq}-{status}/
    ├── report.md        # 执行报告
    ├── screenshots/     # 截图证据
    ├── videos/          # 视频证据
    └── logs/            # 日志证据
```

index.md 维护规范：
```markdown
# TC-{ID} 执行历史

| 执行日期 | 批次 | 结果 | 报告链接 | Issue |
|----------|------|------|---------|-------|
| {date} | {seq} | pass/fail | [report.md](链接) | 无/#N |
```
